1、什么是面向对象？谈谈你对面向对象的理解

对比面向过程，是两种不同处理问题的角度

面向过程更加注重事情的每一个步骤和顺序，面向对象更加注重事情有那些参与者（对象）、及各自需要做什么。

举例：洗衣机洗衣服

面向过程会将任务拆解成一系列步骤（函数）

1、打开洗衣机 2、放衣服 3、放洗衣粉 4、清洗 5、烘干

面向对象会拆出人和洗衣机两个对象

人：打开洗衣机、放衣服、放洗衣粉

洗衣机：清洗、烘干

从上述列子可以看出，面向过程直接高效，而面向对象更易于复用、扩展和维护

面向对象的三大特点：

封装：封装的意义在于明确标识出允许外部使用所有成员函数和数据项，内部细节对外部调用透明，外部调用无需修改或关系内部实现

1、Javabean的属性私有，提供getset对外访问，因为属性的赋值或者获取逻辑只能由Javabean本身决定，而不能由外部胡乱修改。

2、orm框架

操作数据库，我们不需要关系连接是如何建立的，sql是如何执行的，只需要引入mybatis，调用方法即可。

继承：继承基类方法，并作出自己的改变和扩展，子类共性的方法或者属性直接使用父类的，而不需要自己在定义，只需要自己个性化的。

多态：基于对象所属类不同，外部对同一方法的调用，实际执行的逻辑不同。

继承，方法重写，父类引用指向子类对象

弊端：无法调用子类特有的功能

JDK：Java DeveIpment Kit Java开发工具

JRE：Java Runtime Environmrnt Java运行环境

JVM：Java Virtual Machine Java 虚拟机

4、简述 final

最终的

  • 修饰类：表示类不可以被继承
  • 修饰方法：表示方法不可以被子类覆盖，但是可以重载
  • 修饰变量：表示变量一旦被赋值就不可以更改它的值

（1）修饰成员变量

  • 如果final修饰的是类变量，只能在静态初始化块中指定初始化值或者声明该类变量时指定初始值。
  • 如果final修饰时成员变量，可以在非静态初始化块、声明变量或者构造器中执行初始化值。

（2）修饰局部变量

系统不会为局部变量进行初始化，局部变量必须由程序员显示初始化。因此使用final修饰局部 变量时，即可在定义时指定默认值（后面的代码不能在对局部变量在赋值），也不可以指定默认值，而在后面的代码中final变量赋初始值（仅一次）

（3）修饰基本数据类型和引用数据类型

  • 如果是基本数据类型的变量，则其数值一旦在初始化之后便不修改
  • 如果是数据类型的变量，则在其初始化之后便不能让其指向另一个对象，但是引用的值是可以变的。

为什么局部内部类和匿名内部类只能访问局部final变量?

编译之后会生成两个class文件，Test.class 、 Test1.class

  1. 首先需要知道的一点是：内部类和外部类是同处于一个级别的，内部类不会因为定义在方法中就会随着方法的执行完毕就销毁。

  2. 这里就会产生问题：当外部类的方法结束时，局部变量就会销毁了，但是内部类对象可能还存在（只有没有人在引用它时，才会死亡）。这里出现了一个矛盾：内部类对象访问了一个不存在的变量。为了解决这个问题，就将局部变量赋值一份作为内部类的成员变量，这样当局部变量死亡后，内部类仍可以访问它，实际访问的是局部变量的”copy“。这样就好像延长了局部变量的生命周期

  3. 将局部变量复制为内部类的成员变量时，必须保证两个变量是一样的，也就是我们在内部类中修改了成员变量，方法中的局部变量也得随着改变，怎么解决这个问题？

    就将局部变量设置为final，对它初始化后，我们不让你再去修改这个变量，就保证了内部类的局部变量和方法的局部变量的一致性。这实际上也是一种妥协。使得局部变量与内部类建立的拷贝保持一致。

5、String、StringBuffer、StringBuilder区别以及使用场景

String：是final修饰的，不可变，每次操作都会产生新的String对象

StringBuffer和StringBuilder都是在原对象上操作

StringBuffer是线程安全的，StringBuilder是线程不安全的

StringBuffer方法都是synchronized修饰

性能：StringBuilder > > String

使用场景：

经常需要改变字符串时使用后两个

优先使用StringBuilder ，多线程使用共享变量时使用StringBuffer

6、重载和重写的区别

重载：发生在同一个类中，方法名相同，参数类型不同，个数不同，方法返回值访问修饰符可以不同，发生在编译时。

重写： 发生在父子类中，方法名、参数列表必须相同，返回值范围必须小于等于父类，抛出的异常范围小于等于父类，访问修饰符大于等于父类；如果父类方法访问修饰符为private则子类不能重写该方法。

7、接口和抽象类

  • 抽象类可以存在普通成员函数，而接口只能存在public abstract 方法。
  • 抽象类中的成员变量可以是各种类型的，而接口中的成员变量只能是public static final 类型的。
  • 抽象类只能继承一个，接口可以实现多个。

接口的设计目的，是对类的行为进行约束（更准确的说是一种约束，因为接口不可以规定类有什么行为），也就是提供一种机制，可以强制要求不同的类具有相同的行为。它只约束了行为的有无，但不能对如何实现行为进行限制。

而抽象类的设计目的，是代码复用，当不同的类有某些相同的行为（记为行为集合）且一部分的行为的实现方式一致时（A的非真子集，记为B），可以让这些类都派生于一个抽象类。在这个抽象类中实现B，避免让所有的子类来实现B，就达到了代码的复用性，而A减B的部分，留给各个子类来实现，正是A-B在这里面没有实现，所以抽象类不允许实例化（否则当调用A-B时，无法执行）。

使用场景：

当你关注一个事务本质的时候，用抽象类；当你关注操作的时候用接口。

抽象类的功能远超接口，但是定义抽象类的代价高。因为高级语言来说（实际语言来说也是）每个类只能继承一个类，在这个类中你必须继承和编写出其他子类的所以共性。虽然接口上的功能会弱许多，但它只针对一个动作进行描述，而你可以在一个类中实现多个接口，在设计阶段会降低难度。

9、hashCode与equals

hashCode介绍

hashCode（）的作用是获取哈希码，也称为散列码；它实际上是返回一个int整数。这个哈希码的作用是确定该对象在哈希表中的索引位置。hashCode定义在jdk的Objcect.Java中，Java中的任何类都包含hashCode（）函数。撒列表存储的收键值对，它的特点是能根据键快速的查找到对应的值。这其中就运用了散列码。（可以快速的查找对象）

为什么要有hashCode

对象进入HashSet时，HashSet会先计算出对象的hashset值来判断对象加入的位置，看该位置是否有值，如果没有HashSet会假设对象没有重复出现。如果有值会调用equal（）方法来比较这两个值是的相同。如果相同HashSet就不会让其加入其中，如果不相同，就会重新散列到其他位置，这样就减少了equal的次数，大大提高了我们的执行效率。

结论：

  • 如果两个对象相等，则hashCode一定是相等的。
  • 两个对象相等，对两个对象调用equal方法都是返回ture
  • 两个对象有相同的hashCode值，它们也一定相等。
  • 因此equals方法被覆盖过，那么hashCode也必须被覆盖
  • hashCode（）的默认行为是在堆上产生独特值，如果没有重写hashCode，则该class的两个对象无论如何都不相同（即使这两个对象指向相同路径）。

10、ArrayList和LinkedList区别

ArrayList：基于动态数组，连续内存存储，适合下标访问（随机访问），扩容机制：因为数组长度固定，超出长度存储数据时需要创建新的数组，然后将老数组的数据拷贝到新数组中，如果不是尾部插入数据还会涉及到元素移动（往后复制一份，插入新元素），使用尾插法和固定初始容量可以提高性能、甚至超过LinkedList（需要创建大量的node对象）。

LinkedList：基于链表，可以存储在分撒的内存中，适合做数据插入和删除操作，不适合查询，需要逐一遍历。遍历LinkedList必须使用itrator，不能使用for循环遍历，因为每次for循环体内通过git（i）取得某一元素时都需要对list进行重新遍历，性能消耗巨大。

另外不要试图使用indexOf等返回元素索引，并利用其进行遍历，使用indexOf遍历list，当结果为空时会遍历整个列表。

11、HashMap和HashTable有什么区别？其底底层实现是什么？

区别:

  1. HashMap方法没有synchronizd修饰，线程非安全，HashTable线程安全。
  2. HashMap允许key和value为null，而HashTable不允许。

底层实现 ： 数组+链表

jdk8开始链表长度到8，数组长度超过64，链表转换为红黑树，元素以内部类node节点存在

  • 计算数组hash值，二次hash对数组长度取模，对应数组下标。
  • 如果没有产生hash冲突（下标位置没有元素）直接创建node存入数组。
  • 如果产生hash冲突，先进行equal比较，相同则取代该元素，不同则判断链表高度插入链表，链表长度达到8，并且数组长度达到64，自动转换为红黑树，长度低于6则将红黑树转换为链表
  • key为null存在下标0的位置

数组扩容： 跟上面ArrayList一样，如果初始长度不够，就会重新创建一个新的数组，在将老数组的数据拷贝到新数组中。

12、ConcurrentHashMap原理，jdk7和jdk8的区别

jdk7：

数据结构：Reentrantlock+Segment+HashEntry，一个Segment包含一个HashEntry数组，每个HashEntry又是一个链表结构

元素查询：二次hash，第一次hash定位到Segment，第二次hash定位到元素所在链表的头部

锁：Segment分段锁，Segment继承了Reentrantlock，锁定操作的Segment，其他的Segment不受影响，并发数为Segment的个数，可以通过构造函数进行扩容，其他的Segment不会受影响。

get方法无需加锁，volatile保证。

jdk8：

数据结构：synchronized +CAS +Node +红黑树，node的val和next都是用volatile修饰，保证可见性

查找、替换、赋值操作都是使用CAS

锁：锁链表的head节点，不印象其他因素的读写，锁粒度更细，效率更高，扩容时，阻塞所有的读写操作、并发、扩容

读操作无锁：
node的val和next使用volatile修饰，读写线程对该变量相互可见

​ 数组用volatile修饰，保证扩容时被读线程有所感知。

14、什么时字节码？采用字节码的好处是什么？

Java中的编译器和解释器：

Java中引入了虚拟机的概念，即在机器和编码之间加入了一层抽象的虚拟的机器。这台机器在任何平台都提供给编译程序的共同接口。

编译程序只需面向虚拟机，生成虚拟机能够理解的代码，然后由解释器将虚拟机中的代码转化为特点系统的机器码执行。在Java中这种供虚拟机理解的代码叫做字节码（.class文件）它不面向任何特定的的处理器，只面向虚拟机。

每一中平台的解释是不同的，但是实现的虚拟机相同。Java源程序编译后变成字节码文件，字节码由虚拟机解释执行，虚拟机将每一条需要执行的字节码交给解释器，解释器将其翻译成特定机器上的机器码，然后在特定的机器上的运行，这也解释了Java的编译与解释并存的特点。

Java源码—->编译器—->jvm可执行的Java字节码（即虚拟指令）—->jvm —–>jvm中解释器——->机器可执行的二进制机器码—–> 运行程序。

采用字节码的好处:

  • Java语言通过字节码的方式，在一定程度上解决了传统解释型语言效率低的问题，同时也保留了解释型语言可移植的特点。所以Java程序运行时比较高效，而且，由于字节码并不专对一种特定的机器，因此，Java程序无需从新编译即可字其他计算机上进行运行。

15、Java类加载器有那些

jdk自带的有三类加载器：bootstrap ClassLoader、ExtClassLoader、AppClassLoader。

bootstrap ClassLoader是ExtClassLoader的父加载器，默认负责加载%Java_HOME%lib下的jar包和class文件

ExtClassLoader是AppClassLoader的父类加载器，负责加载%Java_HOME%lib下的jar包和class类

AppClassLoader是自定义类加载器的父类，负责加载classpath下的文件。系统加载器，线程上下文加载器继承ClassLoader实现自定义类加载器

16、双亲委派模型

双亲委派模型的好处：

  • 主要为了安全性，避免用户编写的类动态的替换Java的核心类，比如string
  • 同时也避免类的重复加载，因为jvm中区分不同类，不仅仅是根据类名，相同的clss文件被不同的ClassLoader加载就是不同的两个类。

17、Java的异常体系

  • Java中的所有异常都来自于顶级父类Throwable。
  • Throwable下面有两个子类Excption和Error。
  • Error是程序无法处理的错误，一旦遇到该错误，则程序会被迫停止运行。
  • Excption不会导致程序停止运行，又可以分为两个部分RumTimeExceptio运行时异常和CheckedExcption检查异常。RumTimeExceptio常常出现在程序运行过程中，会导致程序当前线程执行失败，CheckedExcption常常发生在编译过程中，会导致程序编译时不通过。
